# PRD

## 1. 任务标识

- 任务名称：演示任务：统一用户导入三端闭环
- 任务目录：`doc/plan/2026-03-26-demo-user-import/`
- 需求来源：`doc/subagents-guide.md` 示例任务
- 负责人：Codex subagents 演示链路

## 2. 核心业务

- 一句话说明这次需求要解决的核心业务问题：让管理员能够在 backend、React、Vue 三端围绕“用户导入”形成一致的计划、实现与验收闭环。
- 为什么现在要做：仓库已存在用户导入能力，适合作为 subagents 文档产出的真实示例，帮助后续需求快速套用标准流程。

## 3. 目标用户与场景

- 目标角色：系统管理员、人事管理员
- 触发场景：管理员需要批量导入用户账号，并选择是否覆盖已存在用户数据
- 当前痛点：三端已有入口，但如果后续要增强交互、导入反馈或校验口径，容易缺少统一的 PRD、设计、技术方案和交付清单

## 4. 主流程

1. 管理员进入用户管理页面并打开“用户导入”对话框
2. 下载模板、填写模板、上传文件，并选择是否覆盖已存在用户
3. 系统校验文件并返回导入结果，管理员确认成功/失败明细

## 5. 功能范围

### 5.1 本次范围

- 把“用户导入”整理成一套可复用的 subagent 示例任务文档
- 明确 backend、React、Vue 三端在该需求下的计划与交付口径
- 为后续真实增强实现提供计划基线

### 5.2 非目标

- 本次不改动数据库结构
- 本次不新增业务系统里的可视化 agent 调度页面/API
- 本次不真正提交用户导入功能代码改动

## 6. 验收契约

- Functional scope：产出一套完整的示例任务文档，覆盖 PRD、DESIGN、TRS、MATERIAL、DELIVERY，并明确三端评估结果
- Non-goals：不执行真实功能开发，不把示例文档伪装成已交付需求
- Exception handling：若示例所引用的仓库能力与现状不一致，应在文档中标记假设与阻塞
- Required logs or verification evidence：至少记录计划中的验证命令和“本示例未实际执行”的事实
- Rollback trigger or rollback conditions：若示例文档误导读者认为该需求已真实上线，则应撤回并改写为“演示任务”

## 7. 验收标准

- 五份示例文档齐全且位于 `doc/plan/2026-03-26-demo-user-import/`
- `TRS.md` 和 `DELIVERY.md` 都明确说明 backend、React、Vue 三端状态
- `DELIVERY.md` 显式标注“示例文档，未执行代码实现”

## 8. 风险与阻塞

- 示例任务如果写得过于抽象，后续团队难以直接复用
- 示例任务如果写得像真实交付，可能造成误解
